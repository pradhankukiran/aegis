# Aegis — security audit (Phase 6, Wave 6a)

Read-only review of `src/lib/` and `src/app/api/` against the threat model in
`aegis-plan.md` §8. No code changes were made; the tech-debt agent will apply
fixes based on this report.

## 1. Methodology

Read every `.ts` file in `src/lib/` (about 60 source modules, excluding tests)
and the one route in `src/app/api/pinata/upload-url/`. Crypto primitives
(`crypto/symmetric.ts`, `crypto/kdf.ts`, `crypto/random.ts`, `crypto/schnorr.ts`,
`crypto/halo-prf.ts`, `crypto/timelock.ts`, `crypto/shamir.ts`) read end-to-end.
Each feature's envelope, transport-bridge, storage, and types were inspected;
hooks were skimmed for seckey handling and randomness sources. Cross-cutting
greps verified: all `encryptBytes` callers attach AAD; all `schnorr.sign`
callers digest a canonicalized payload; every IndexedDB DB name is unique;
no `dangerouslySetInnerHTML`, no `innerHTML =`, no `eval`/`new Function`;
`Math.random()` is used only for non-security identifiers; all `process.env`
references categorized (server-only vs `NEXT_PUBLIC_*`). Wire formats for
beacon cancellation, witness anchor, and quorum ballot signing inspected
against §8 of the plan. The two stub `auth/` files (`csrf.ts`, `jwt.ts`) were
traced for callers — both are dead code (Hermetic port artifacts).

## 3. Per-feature notes

**crypto** — Solid. Single libsodium init via `getSodium()`, sane Argon2id presets (light/balanced/strong), XChaCha20-Poly1305 with random 24-byte nonces (collision-safe), HKDF info strings well-domain-separated. Schnorr Σ-protocol module is unused dead code; either delete it or extend its API to bind a context (SEC-017). `halo-prf.ts` looks well-engineered but isn't consumed anywhere yet — it's a forward-looking Halo passkey scaffold.

**identity** — Master keypair in IDB unencrypted, which matches Hermetic's pattern. The portable-export format (`exportIdentity`) is unencrypted base64url of the seckey (SEC-010). The master seckey is passed to `nip44.v2.utils.getConversationKey` (Nostr DM), `hkdf` (SSB seed, Scribe master key), and `schnorr.sign` (Witness, beacon-cancel, Quorum-vote). No stray copies; no plaintext logs.

**transport** — Three-network facade is solid. `publish` is per-network resilient; `subscribe` dedup is correct; idempotent unsubscribes everywhere. Issues: Matrix access-token in localStorage (SEC-002), SSB DM fallback in plaintext (SEC-003), bundled registration token (SEC-004), author-filter not honoured for Matrix/SSB (SEC-006). The MXID truncation (24 hex chars, SEC-012) is a documented architectural call.

**pinata** — Browser side requests signed URLs and uploads ciphertext; server side holds the JWT and never echoes it. The route is correct in isolation but unauthenticated (SEC-001).

**herald** — DM in/out via `transport.subscribeDM` / `transport.directMessage`. Messages stored as plaintext in IDB (`aegis-herald.messages`), which is the documented trade. No XSS concerns (React text rendering). Conversations keyed by raw `from` string, so a peer's Nostr-origin and Matrix-origin DMs land in two rows until the directory lookup lands (SEC-012, documented).

**scribe** — Two-tier envelope (master key from HKDF of seckey, per-note key wrapped under master). AAD `aegis:notes:v=1` on both wrap and content. Title left in plaintext (SEC-013, documented). Save markers leak metadata to SSB (SEC-014). Y.js CRDT module exists but isn't bound to a transport yet.

**atlas** — Per-tick position fix → JSON-serialize `{type, fix}` → `transport.directMessage(member, plaintext)`. When SSB fallback fires, coordinates leak (SEC-003). No encryption layer above what `directMessage` provides. Bounded per-peer position log (50 fixes), proper bounds-check on incoming lat/lon.

**witness** — BIP-340 Schnorr over `sha256(canonicalize({hash, ts}))`. Anchor verification is byte-deterministic. Hash route validates input. Cross-network verification subscribes for 4 seconds then closes — reasonable. Replay-of-old-ts is by-design (SEC-011).

**beacon** — Two-layer release (fast watchdog publish + slow tlock-anchored release). Signed cancellation correctly covers `{beaconId, ts}` but doesn't bound `ts` (SEC-005). `unwrapKeyHex` stored in IDB alongside CID — documented trade-off (the key WILL leave, this just makes fast-path instant). Watchdog status-write-before-publish prevents duplicate fires.

**quorum** — Sign-then-seal pattern is correct: BIP-340 over `sha256(canonicalize({pollId, optionIndex, voter, nonce}))`, then tlock to drand round. `unsealVote` verifies sig against claimed voter pubkey. PollMeta wire shape lacks a signature over the metadata (SEC-009). Compound IDB key `[pollId, voter]` enforces one-ballot-per-voter at the storage layer.

**crucible** — Source side generates ephemeral keypair, ECDH(ephemeral, newsroom) → HKDF → CEK, encrypts under XChaCha20 with AAD `aegis:crucible:v=1`, uploads to Pinata, publishes a pointer event. `seckey.fill(0)` and `cek.fill(0)` in `finally` block — verified. Newsroom side filter accepts both 64- and 66-char hex `to` values. No persistence on the source side, by design.

**auth** — Both files (`csrf.ts`, `jwt.ts`) are unused dead code from the Hermetic port (SEC-007, SEC-008). `csrf.ts` correctly fail-closed.

## 4. Open architectural debt

Catalogued from comments + plan §10 + audit observations. Severity is independent of the SEC-* findings table.

- **MXID ↔ pubkey directory** (`transport/index.ts:117-126`, `herald/transport-bridge.ts:9-21`) — Matrix DM bridge cannot recover full pubkey from a received MXID (24-char hex truncation is one-way); same problem for SSB feed ids. Until a directory exists, the same peer appears as up to three separate "from" identities. **Severity: medium. Feasibility: medium** — needs a tiny `aegis-directory` event or NIP-65-style relay list.
- **NIP-78 per-hash `d`-tag for Witness** (`witness/verify.ts:9-39`) — current verification uses a cross-network subscribe with a 4s wait window. A per-hash addressable kind (or a separate kind for anchors with content-tagged `d`) would give O(1) `ids:[id]` lookup. Phase 4 open question, documented inline. **Severity: low. Feasibility: small.**
- **SSB private boxes** (`transport/ssb.ts`, `transport/index.ts:668-673`) — DMs over SSB are unencrypted; see SEC-003. libsodium `crypto_box_seal` would close it without needing a homeserver-side change. **Severity: high. Feasibility: medium.**
- **NostrTransport.publish event id round-trip** (`witness/anchor.ts:30-38`, `transport/index.ts:435-445`) — `SimplePool.publish` doesn't surface the wire event id, so `publishResults[]` for nostr is missing `eventId`, and Witness can't do an O(1) re-lookup later. Documented as Phase 4 open question. **Severity: low. Feasibility: small** — recompute the id from the signed event canonically.
- **Paillier homomorphic tally for Quorum** (`quorum/types.ts:38-46`) — current commit-reveal exposes each individual vote post-close. Plan §10 deferred this to Phase 6+. **Severity: low** (no immediate threat — current model is documented). **Feasibility: large.**
- **Anti-Sybil for polls** (general) — Quorum's `voters: []` (empty = open poll) accepts any well-signed ballot, so a Sybil attacker mints arbitrary identities and votes N times. **Severity: medium for "anyone may vote" polls. Feasibility: large** — would need a proof-of-personhood layer (BrightID-like) or per-poll deposit/cost.
- **Scribe title sealing** (SEC-013) — title in plaintext in IDB. **Severity: low. Feasibility: small** — wrap a "metadata key" eagerly on identity unlock and seal titles under it.
- **Cross-device note sync** (`scribe/feed.ts` and the live-infra notes throughout) — Save markers go to SSB in plaintext; actual content sync via Pinata isn't wired. **Severity: low (functionality, not security). Feasibility: medium.**
- **Identity recovery / Shamir backup** (plan §10) — no UI surface exists yet; users with a wiped IDB lose everything. **Severity: high for production use, low for portfolio scope. Feasibility: medium** (Hermetic's Switch pattern is portable).
- **Cross-signing / SAS device verification for Matrix** (`transport/matrix.ts:34-37`) — single-device only. **Severity: medium. Feasibility: large.**
- **Matrix access token at-rest** (SEC-002) — localStorage. **Severity: high. Feasibility: medium.**
- **Quorum poll metadata not signed** (SEC-009) — peer can forge ownership claims. **Severity: medium. Feasibility: small.**
- **Beacon cancellation ts window** (SEC-005) — no bound on `ts`. **Severity: medium. Feasibility: small.**
- **API route hardening** — Pinata route is unauthenticated (SEC-001); registration token bundled into client (SEC-004). **Severity: high. Feasibility: small.**
- **Wire-payload length caps** (SEC-015) — every `projectXxx` should cap incoming string lengths. **Severity: low. Feasibility: small.**
- **Dead code in `auth/`** (SEC-007, SEC-008) — misleading; remove. **Severity: low. Feasibility: small.**
- **Halo PRF / WebAuthn** (`crypto/halo-prf.ts`) — present, untested in any feature path. **Severity: info. Feasibility: large** to integrate.
- **Schnorr Σ-protocol unused** (SEC-017) — present without a context-binding API. **Severity: low. Feasibility: small.**

## 5. Recommended fix order

1. **SEC-001** — wire `assertSameOrigin` into `POST /api/pinata/upload-url`. This is the only Aegis-shipped HTTP surface; leaving it open lets anyone burn the Pinata quota.
2. **SEC-004** — move `MATRIX_REGISTRATION_TOKEN` off the client bundle into a server route. Without this, anyone with the deployment URL owns the homeserver's registration capability.
3. **SEC-002** — encrypt or HttpOnly-cookify the Matrix access token. The combo of SEC-002 + an XSS lapse leaks the entire Matrix history.
4. **SEC-003** — gate or seal the SSB DM fallback. Atlas users sharing location have no warning that an "all Matrix and Nostr unreachable" condition leaks their coordinates to a public feed.
5. **SEC-009 + SEC-005** — close the two replay/forgery gaps in Quorum and Beacon. Both are one-page diffs in `seal.ts` / `cancel.ts` plus a tiny `verify*` change.
